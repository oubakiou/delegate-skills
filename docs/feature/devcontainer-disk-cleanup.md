# devcontainer ディスククリーニング定型化 設計・実装計画

[![MKDN](https://img.shields.io/badge/MKDN-review-red?style=for-the-badge)](https://mkdn.review/?url=https%3A%2F%2Fraw.githubusercontent.com%2Foubakiou%2Fdelegate-skills%2Frefs%2Fheads%2Fmain%2Fdocs%2Ffeature%2Fdevcontainer-disk-cleanup.md)

[development.md のセットアップ節](../design/development.md#セットアップ)に対応し、devcontainer のコンテナディスクが再生成可能なキャッシュで満杯になり全ツールが停止する事象（ENOSPC）を、定型スクリプトで予防・回復するための設計判断と実装手順をまとめる。完了後は development.md に永続情報を移し、本ファイルは archive する。

本計画の前提は、コンテナディスク逼迫の原因が 2 層に分かれるという実測である。コンテナ内から回収できるのは VS Code server / npm 等のキャッシュ約 12GB のみで、残る大半（実測時約 50GB）はホスト Docker Desktop VM 側にあり、コンテナ内からは操作できない。本計画はコンテナ内の回収を定型化し、VM 側はホスト操作の案内出力に留める。

## 1. 対応スコープ

| 要件                                                            | 開始時の状態                                                                                                     | 完了条件                                                                                                                                                                               | 最終状態 | 状態   |
| --------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------- | ------ |
| [MUST] 再生成可能キャッシュを冪等に回収するスクリプトを提供する | 手動調査 + 個別 `rm` / `npm cache clean` の非定型作業（2026-07 の障害対応で実施）                                | `scripts/clean-devcontainer-disk.sh` が削除対象の列挙・実行・回収量報告を行い、連続実行しても安全である                                                                                | 未実装   | 未着手 |
| [MUST] 使用中リソースを誤削除しない fail-safe を持つ            | 使用中 VS Code server 世代の特定は手動（`ps` 目視）                                                              | 使用中リソース（VS Code server 世代、稼働中の cursor-agent / codex）をカテゴリ別に動的検出し、検出不能時は該当カテゴリの削除を skip して報告する。`/vscode` 不在環境では何も削除しない | 未実装   | 未着手 |
| [MUST] 毎起動時の自動実行経路を持つ                             | 掃除は障害発生後の手動対応のみ。`.devcontainer/devcontainer.json` は `postCreateCommand`（初回のみ）しか持たない | devcontainer `postStartCommand` から毎起動時に閾値チェック付きで呼ばれ、閾値未満なら何もしない。`local_setup.sh`（初回セットアップ）からも同条件で呼ぶ                                 | 未実装   | 未着手 |
| [SHOULD] コンテナ内で解決できない分をユーザーに案内する         | VM 側使用分の存在が誤診（`.temp/` 犯人説）を招いた                                                               | スクリプト出力が「コンテナ内回収可能量」と「ホスト側 `docker system prune` が必要な旨」を区別して表示する                                                                              | 未実装   | 未着手 |
| [SHOULD] 削除せず対象だけ確認できる dry-run を持つ              | なし                                                                                                             | `--dry-run` が削除候補と推定回収量のみ表示し、ファイルシステムを変更しない                                                                                                             | 未実装   | 未着手 |

スコープ外:

- ホスト Docker Desktop VM 側の掃除（`docker system prune`、仮想ディスクの reclaim / 上限設定）: コンテナ内から操作不能。スクリプトは案内メッセージの出力までを責務とする
- `.temp/` / delegate run スクラッチの掃除: リポジトリはホストマウント側にありコンテナディスクを圧迫しない。既存の `DELEGATE_RUN_RETENTION_DAYS` が所有する
- `~/.vscode-server/extensions`（拡張本体）の削除: 使用中の可能性があり、キャッシュではないため対象にしない
- cron / 常駐監視: devcontainer は再作成が頻繁で持続性が保証されず、起動時ワンショットで十分（§5.b）

### 2.2 実測（2026-07-21 障害時）

コンテナディスク `/dev/vda1` 71GB が使用率 100%（空き 2.4MB）となり、Bash ツール実行を含む全書き込みが停止した。リポジトリのある `/workspaces` はホストマウントで無関係だった。

| 対象                                              | 実測サイズ | 分類                                 | 対応                       |
| ------------------------------------------------- | ---------- | ------------------------------------ | -------------------------- |
| `/vscode/vscode-server/extensionsCache`（126 件） | 6.7GB      | 再生成可能キャッシュ（root 所有）    | sudo で削除                |
| `/vscode/vscode-server/bin/linux-arm64`（8 世代） | 3.8GB      | 使用中 1 世代以外は不要（root 所有） | 使用中を `ps` で特定し残存 |
| `~/.npm/_cacache`                                 | 1.6GB      | 再生成可能キャッシュ                 | `npm cache clean --force`  |
| `~/.vscode-server/extensionsCache`                | 342MB      | 再生成可能キャッシュ                 | 削除                       |
| `~/.local/share/cursor-agent/versions`（3 世代）  | 592MB      | 旧世代は不要                         | 今回は未対応（本計画対象） |
| `~/.codex/.tmp`                                   | 88MB       | 一時ファイル                         | 今回は未対応（本計画対象） |
| ホスト Docker Desktop VM 側                       | 約 50GB    | コンテナ内から不可視・操作不能       | ホスト側 prune の案内のみ  |

手動対応の回収量は約 11.6GB（使用率 100% → 83%）。キャッシュは VS Code / 拡張の更新のたびに再蓄積するため、一度きりの掃除では再発する。

## 3. 設計の中核

### 3.1 スクリプト構成

| 構成要素                             | 内容                                                                                              | 配置 / 寿命                                |
| ------------------------------------ | ------------------------------------------------------------------------------------------------- | ------------------------------------------ |
| `scripts/clean-devcontainer-disk.sh` | 削除候補の列挙 → fail-safe 判定 → 削除実行 → 回収量と VM 側案内の報告。`--dry-run` / 閾値引数付き | リポジトリ常設。手動・起動時の両方から呼ぶ |
| devcontainer `postStartCommand`      | 毎起動時、使用率が閾値（既定 90%）以上のときのみ実行                                              | `.devcontainer/devcontainer.json` に追加   |
| `local_setup.sh` からの呼び出し      | 初回セットアップ時に同条件で実行（`postCreateCommand` 経路）                                      | 既存セットアップフローに 1 行追加          |

削除対象は §2.2 の「再生成可能キャッシュ」「旧世代」列に限定し、候補列挙と削除実行をスクリプト内で分離する。dry-run は列挙のみ実行する。

閾値判定は `df -P /` が報告するコンテナ root filesystem（overlay）の使用率で行い、ホストマウントの `/workspaces` は測定対象にしない（2026-07 の障害では両者の混同が誤診を招いた）。

### 3.2 fail-safe 規則

- 使用中 VS Code server 世代は `ps` の `vscode-server/bin/<arch>/<commit>` パターンから動的検出する。検出できない場合、server 世代カテゴリの削除は行わず skip を報告する（誤削除すると IDE 接続が切れるため fail-closed）
- `/vscode` 配下は root 所有のため非対話 `sudo -n` を前提とし、使えない環境では該当カテゴリを skip して報告する
- `/vscode/vscode-server` が存在しない環境（devcontainer 以外）では削除対象なしとして即終了する
- cursor-agent 旧世代は、cursor-agent プロセスが稼働していないことを確認した上で最新 1 世代を残して削除する。稼働中、プロセス確認不能、世代判別不能のいずれの場合もカテゴリごと skip する
- `~/.codex/.tmp` は codex プロセスが稼働していないことを確認した上で削除する。稼働中またはプロセス確認不能の場合は skip する

### 3.3 終了コード契約とテスト注入境界

起動フックから無人実行されるため、終了コードを契約として固定する。

| 終了コード | 意味                                                                                      |
| ---------- | ----------------------------------------------------------------------------------------- |
| 0          | 回収成功・no-op（閾値未満 / 候補なし）・カテゴリ skip を含む正常系。skip 内容は出力で報告 |
| 2          | 引数エラー                                                                                |
| その他非 0 | 予期しない失敗                                                                            |

`postStartCommand` / `local_setup.sh`（`set -euo pipefail`）からの呼び出しは、スクリプトの失敗がコンテナ作成・起動をブロックしない形（失敗を警告表示に留める）にする。

テストのため、走査 root ディレクトリ・process listing・sudo 実行コマンドは引数または環境変数で差し替え可能にし、fake ディレクトリツリー上で候補列挙と削除実行の双方を検証できるようにする（§6）。

## 4. 実装ステップ

### Step 1: (未着手) スクリプト実装

- `scripts/clean-devcontainer-disk.sh` に §3 の列挙 / fail-safe / 削除 / 報告を実装
- `--dry-run` と閾値引数（`--threshold <pct>`、既定は無条件実行）を実装
- 出力にコンテナ内回収量とホスト側 `docker system prune` 案内を分けて表示

成果物: 冪等な掃除スクリプト + dry-run

### Step 2: (未着手) 毎起動実行の統合

- `.devcontainer/devcontainer.json` に `postStartCommand` を追加し、毎起動時の閾値チェック付き呼び出し（既定 90%）を行う（現状は `postCreateCommand` のみで、コンテナ再起動時に掃除が走らない）
- `local_setup.sh`（初回セットアップ）にも同じ呼び出しを追加
- どちらの経路もスクリプト失敗でコンテナ作成・起動をブロックしない（§3.3）

成果物: 毎起動の自動予防経路

### Step 3: (未着手) テストと検証

- 列挙ロジックの golden 検証を Vitest 契約テストに追加（§6）
- 実 devcontainer で dry-run → 実実行 → 再実行（冪等性）→ `vp check` / `npm test` を確認

成果物: 契約テスト + 実機検証記録

### Step 4: (未着手) development.md 反映

- development.md のセットアップ節に掃除コマンドと ENOSPC 時の診断順（`df` でコンテナ側と `/workspaces` を区別 → 本スクリプト → ホスト側 prune）を追記

成果物: development.md 更新

### Step 5: (未着手) typescript-agent-package-template への反映と archive 化

- [typescript-agent-package-template](https://github.com/oubakiou/typescript-agent-package-template) に確立したスクリプトと起動時統合を移植する（テンプレート側の `local_setup.sh` / devcontainer フック構成（`postCreateCommand` / `postStartCommand` の有無）・ドキュメント規約の差異を確認して調整）
- テンプレート側ドキュメントに ENOSPC 診断順を追記する
- 本ドキュメントを `docs/archive/devcontainer-disk-cleanup.archive.md` に移す

成果物: テンプレートリポジトリへの PR + archive（archive 化はユーザー確認後）

## 5. 設計判断

### a. 実現形態

| 候補                                  | 採用 | 理由                                                                                              |
| ------------------------------------- | ---- | ------------------------------------------------------------------------------------------------- |
| **リポジトリ常設の shell スクリプト** | ✓    | 削除対象は判断の余地がないキャッシュで決定論的。`local_setup.sh` という運用スクリプトの前例に沿う |
| Claude Code skill（`/clean-disk`）    | ✗    | LLM を挟む必然性がなく、token を消費する。決定論的な処理は skill にしない方針と整合しない         |
| エージェントへの都度依頼（非定型）    | ✗    | 障害のたびに調査からやり直しになり、`.temp/` 犯人説のような誤診も再発し得る                       |

### b. 実行トリガー

| 候補                                   | 採用 | 理由                                                                             |
| -------------------------------------- | ---- | -------------------------------------------------------------------------------- |
| **起動時ワンショット（閾値チェック）** | ✓    | devcontainer の寿命と一致し、閾値未満なら no-op でコストゼロ                     |
| cron / 常駐監視                        | ✗    | コンテナ再作成で消え、持続性の保証に追加の仕組みが要る。過剰                     |
| 手動実行のみ                           | ✗    | 満杯になってからでは Bash 自体が失敗し、実行手段を失っている可能性がある（実測） |

### c. 削除対象の範囲

| 候補                                        | 採用 | 理由                                                                                     |
| ------------------------------------------- | ---- | ---------------------------------------------------------------------------------------- |
| **再生成可能キャッシュ + 未使用旧世代のみ** | ✓    | 誤削除リスクを構造的に排除でき、無人実行（起動時）に耐える                               |
| 拡張本体・使用中世代も含む積極削除          | ✗    | IDE 接続断・拡張再インストールの実害があり、回収量の上積みも小さい                       |
| ホスト VM 側も対象に含める                  | ✗    | コンテナ内から操作不能（docker socket は空の docker-in-docker daemon）。案内出力に留める |

## 6. テスト方針

### 自動テスト

§3.3 の注入境界（走査 root / process listing / sudo コマンドの差し替え）を使い、fake ディレクトリツリー上で契約テスト（Vitest）を行う。

- 候補列挙（`--dry-run` の golden）
  - 使用中 server 世代が候補から除外される
  - 使用中世代を検出できない場合に server カテゴリ全体が skip される
  - cursor-agent / codex プロセス稼働中に該当カテゴリが skip される
  - `/vscode/vscode-server` 不在時に候補ゼロで正常終了する
- 削除実行（fake tree 上）
  - 対象カテゴリだけが削除され、対象外（拡張本体・使用中世代）が保持される
  - 空白を含むパスが安全に削除される
  - 連続実行で候補が空になる（冪等性）
- 閾値と終了コード
  - 既定閾値 90% に対し 89% / 90% / 91% の境界で実行有無が正しい
  - no-op / カテゴリ skip が exit 0、引数エラーが exit 2 になる
- 実環境（root 所有の `/vscode`、実 `ps`）に依存する検証のみ手動確認に残す

### 手動確認

- [ ] 実 devcontainer で `--dry-run` の候補列挙が §2.2 の分類と一致する
- [ ] 実実行後に `df` の空きが増え、IDE 接続と拡張が正常なまま
- [ ] 直後の再実行が no-op になる
- [ ] `sudo -n` 不可を模擬した場合に `/vscode` カテゴリが skip 報告され、exit 0 で終了する
- [ ] コンテナ再起動で `postStartCommand` 経路が実行される（閾値未満なら no-op ログのみ）
- [ ] `vp check` / `npm test` が通る

## 7. 受け入れ基準

- §1 の MUST 要件をすべて満たす
- 使用中の VS Code server 世代・拡張本体・稼働中 cursor-agent / codex のリソースがいかなる分岐でも削除されない
- 毎起動（`postStartCommand`）経路は閾値未満で no-op、閾値超過で無人回収に成功する
- skip / no-op を含む正常系が exit 0 で終了し、起動フックを失敗させない（§3.3 の契約どおり）
- スクリプト出力がコンテナ内回収量とホスト側対応の必要性を区別して報告する
- development.md と実装が一致している
- typescript-agent-package-template への反映 PR が作成されている

## 8. 想定リスクと回避策

| リスク                                                       | 回避策                                                                                 |
| ------------------------------------------------------------ | -------------------------------------------------------------------------------------- |
| 使用中 server 世代の検出漏れによる IDE 切断                  | 検出不能時は削除ではなく skip（fail-closed）。検出パターンを契約テストで固定           |
| VS Code 側のディレクトリ構成変更でパスが陳腐化する           | パス不在は候補ゼロの正常系として扱い、エラーにしない。回収ゼロが続く場合に構成を再調査 |
| 満杯時にスクリプト自体が書き込み失敗で動かない               | スクリプトは一時ファイルを作らず列挙・削除のみ行う。閾値起動で満杯前に回収する         |
| ホスト VM 側が原因の場合に「掃除したのに直らない」と誤認する | 出力で VM 側約 50GB 相当の存在とホスト側 prune の必要性を明示する                      |
| カテゴリ skip や部分失敗が起動フック全体を失敗させる         | §3.3 の終了コード契約で skip を exit 0 に固定し、フック側も失敗を警告表示に留める      |
| テンプレート側と本リポジトリでスクリプトが乖離する           | Step 5 で移植時の調整点を PR に記録し、以後の改善は双方の issue で相互参照する         |
